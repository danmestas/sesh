# Synadia Protocol v0.4 — A2A feature-parity design

## Executive summary

This document specifies v0.4 of the Synadia Agent Protocol — an additive
extension to the wire format used by the sesh mesh that closes the
feature-parity gap with the Google/Linux Foundation **A2A v1.0** protocol.

Once shipped, a sesh mesh can be addressed by any external A2A-compliant
client (LangChain, CrewAI, custom agents, stock `a2a-python`/`a2a-go` SDK
consumers) via a dedicated **HTTPS+JSON-RPC gateway binary** (`sesh-shim`)
that translates A2A on the wire to NATS subjects and JetStream KV
operations on the mesh.

The design has four cornerstones:

1. **Hybrid versioned envelopes.** A v2 envelope shape coexists with the
   existing v1 envelope on a separate subject prefix
   (`agents.prompt.v2.*`). Adapters opt in to v2 individually; v1
   continues to work unchanged.
2. **Companion KV buckets via sesh-ops library.** Sesh-ops is and
   remains a Go library + reference CLI over JetStream KV — *not* a
   long-running daemon. v0.4 extends the sesh-ops library with new
   **Messages** and **Artifacts** packages (siblings to the existing
   Task package), plus an Object Store helper for large file parts.
   The Task schema gains one new status (`rejected`) and one optional
   discriminator field (`BlockedReason`). The shim imports sesh-ops
   and talks directly to JetStream KV; no RPC tier sits between them.
3. **HTTPS gateway as a separate binary.** `sesh-shim` is a stateless Go
   binary in `sesh/cmd/sesh-shim/` that holds A2A's HTTPS+JSON-RPC
   surface, JCS+JWS signing of AgentCards, SSE bridging, push
   notification delivery, and authentication. Mesh-internal tooling is
   unaffected by its presence or absence.
4. **Adapter-supplied AgentCards.** Each sesh-channels adapter
   contributes the rich, per-agent portion of its A2A AgentCard
   (skills, description, icon) via a new `agents.card.get.*` request
   handler. The shim merges adapter contributions with mesh-wide
   defaults and per-deployment configuration, then signs the result.

**Transport binding decision**: this document specifies the A2A
**JSON-RPC binding**. Method names below use the JSON-RPC PascalCase
convention from the a2a-go SDK (`SendMessage`, `GetTask`,
`SubscribeToTask`, etc.). The REST binding (`POST /message:send` etc.)
is out of scope for v0.4; it can be added later as a thin alternative
gateway pointing at the same shim core.

The implementation spans three repos and five rollout phases, all
**zero-migration and additive**. No v0.3 deployment requires changes to
continue working. Mixed v0.3/v0.4 meshes are an explicit supported
configuration.

## Background and motivation

### What sesh currently has (v0.3)

The Synadia Agent Protocol v0.3 — the wire format used by the sesh mesh
today — exposes these subjects:

| Subject | Direction | Purpose |
|---|---|---|
| `agents.prompt.<agent>.<owner>.<name>` | client→adapter | route a prompt to one agent |
| `agents.prompt.<machine>.<project>.<session>.<role>[.<inst>]` | client→adapter | coord-routed prompt (sesh #91/#94) |
| `agents.hb.<agent>.<owner>.<name>` | adapter→bus | per-agent heartbeat |
| `agents.status.<agent>.<owner>.<name>` | adapter→bus | per-agent status updates |
| `agents.attachments.<agent>.<owner>.<name>` | adapter→bus | file attachment notices |
| `$SRV.INFO.agents` / `$SRV.PING.agents` | client→adapter | NATS micro service discovery |
| `agents.report.<machine>.<project>.<session>[.<role>[.<inst>]]` | adapter→bus | coord-aware status reports |

This surface supports the internal sesh use case (driving long-running
LLM adapters from a CLI on the same NATS mesh) but does not match A2A's
expectations of an agent endpoint.

### What A2A requires (v1.0)

A2A v1.0 is the merged successor to ACP, under the Linux Foundation as
of 2026. Its on-the-wire surface is HTTPS+JSON-RPC with SSE for
streaming. The data model contains 18 named objects (Message, Part,
Task, Artifact, AgentCard, …) and the following RPC methods (per
`a2a-go/a2asrv/handler.go`):
`SendMessage`, `SendStreamingMessage`, `GetTask`, `ListTasks`,
`CancelTask`, `SubscribeToTask`,
`CreateTaskPushNotificationConfig`, `GetTaskPushNotificationConfig`,
`ListTaskPushNotificationConfigs`, `DeleteTaskPushNotificationConfig`,
`GetExtendedAgentCard`. Task state is one of nine values
(`TASK_STATE_SUBMITTED`, `TASK_STATE_WORKING`, …,
`TASK_STATE_UNSPECIFIED`); AgentCards are JCS-canonicalized and
JWS-signed.

A complete gap inventory lives at
`docs/proposals/2026-05-24-a2a-feature-parity-shim-map.md`; the headline
is that ~60% of A2A surfaces have no v0.3 analogue. The largest gaps
are: structured Task lifecycle (sesh-ops covers ~80% of this), per-task
message history (none in v0.3), per-task artifact assembly (none),
AgentCard (none), HTTPS gateway (none), push notifications (none),
signed discovery (none).

### Why now / why this shape

External A2A consumers are the gating audience — internal sesh tooling
already has its own observability and routing primitives via `sesh
mesh`, `$SRV.INFO`, and the existing coord subjects. The minimal
addition that unlocks external consumption is an HTTPS gateway plus
just-enough new wire structure to round-trip A2A's data model without
lossy translation.

The "just-enough" criterion drove three design choices:

- **No big-bang v0.4 envelope replacing v0.3.** v1 prompts continue
  unchanged on their existing subject. v2 prompts ride a new
  `agents.prompt.v2.*` subject prefix so queue-group routing cleanly
  separates traffic.
- **Sesh-ops as substrate, not sesh.** Sesh-ops already has a robust,
  CAS-protected, watchable Task store. Adding Messages + Artifacts as
  sibling buckets is far cheaper than reinventing that infrastructure
  inside sesh.
- **Shim as a separate binary.** A2A's HTTPS surface, auth, signing,
  SSE, and push notifications are orthogonal to the mesh's NATS
  semantics. Embedding them into adapters or into `sesh` itself would
  conflate concerns. A standalone binary lets ops harden, scale, and
  upgrade the gateway independently.

### Three-repo work distribution

| Repo | v0.4 deliverable | Approx % of total work |
|---|---|---|
| **sesh** | Design doc (this), `cmd/sesh-shim/` binary, `sesh mesh card` debug subcommand, envelope v2 spec | 40% |

**Note on sesh-ops topology**: sesh-ops is a Go library (with a
reference CLI). Both `sesh-shim` and `sesh-channels` adapters import
it for typed access to Task/Messages/Artifacts schema and CAS-protected
operations. There is no `sesh-ops` daemon and no NATS RPC tier
between consumers and JetStream KV.

### Scope as the cross-cutting key

Sesh-ops introduced the `{scope-kind, scope-id}` tuple as its bucket
keying scheme: every Task lives in a bucket named
`sesh_tasks_<scope-kind>_<scope-id>`. v0.4 adopts the same convention
for Messages, Artifacts, AgentCards, and notification configs. Scope
kinds in active use:

| scope-kind | Typical scope-id | Examples |
|---|---|---|
| `hub` | hub instance name | shared NATS hub serving multiple users |
| `project` | project label | a sesh project (often 1:1 with a git repo) |
| `session` | sesh session label | one editing/coding session |
| `workflow` | workflow run ID | a multi-step orchestration |
| `agent` | agent identity | per-agent private state |

The shim's principal-to-scope authorization happens here: a Bearer
token might grant a principal access to `scope: session:rc-dev` but not
to `scope: session:other-team`. The shim refuses to read from or write
to scope buckets the principal isn't authorized for.

## Section 2 — Envelope v2

### Why versioned envelopes (not a single in-place evolution)

A2A's Message data model is richer than the v0.3 prompt body:

- A2A Messages have an explicit `messageId` (caller-supplied or
  adapter-supplied ULID), `taskId`, `contextId`, `role` (user|agent),
  `parts[]` (mixed text/file/data), `metadata`, `extensions`, and
  `referenceTaskIds`.
- v0.3 prompts are unstructured strings (or simple JSON objects with a
  `text` field).

Two strategies were considered:

- **Big-bang in-place evolution** — change the v0.3 envelope to be a
  superset. Risk: every adapter and every CLI caller breaks until they
  all upgrade simultaneously.
- **Versioned envelopes** — v1 stays at its existing subject, v2 rides
  a separate subject. Adapters opt in independently.

This document uses versioned envelopes, with subject-level
discrimination via `agents.prompt.v2.*`.

### v2 envelope shape

A v2 prompt body is a **byte-for-byte A2A Message** (per
`a2a-go/a2a/core.go:199`). No sesh-internal fields appear in the body
itself. Sesh routing context travels as NATS message headers during
transit (see "Routing metadata" below).

```json
{
  "messageId": "01HX8K2N7QGZ9Y3VRWMTBV5DPE",
  "taskId": "01ARYZ6S41TSRCOSKF2FVG69G2",
  "contextId": "session:rc-dev",
  "role": "user",
  "parts": [
    {"text": "Review this PR diff and find security issues.",
     "mediaType": "text/plain"},
    {"url": "obj://sesh_objects_session_rc-dev/01ARYZ.../msg/0",
     "mediaType": "text/x-diff",
     "filename": "pr-1234.diff"}
  ],
  "extensions": [],
  "referenceTaskIds": [],
  "metadata": {}
}
```

**Part shape note**: Parts have **no `kind` discriminator** field. Per
the a2a-go SDK (`a2a/core.go:582-756`), Parts are flat objects where
the presence of `text` / `data` / `url` / `raw` determines the variant:

```json
{"text": "...",        "mediaType": "text/plain"}              // TextPart
{"data": {"k": "v"},   "mediaType": "application/json"}        // DataPart
{"url": "https://...", "mediaType": "video/mp4",
                       "filename": "demo.mp4"}                 // FilePart (by reference)
{"raw":  "<base64>",   "mediaType": "application/pdf",
                       "filename": "doc.pdf"}                  // FilePart (inline bytes)
```

**Required-field note**: `messageId`, `role`, and `parts` are required;
`taskId`, `contextId`, `extensions`, `referenceTaskIds`, and `metadata`
are optional (omitted when not relevant). The first message of a new
conversation omits both `taskId` and `contextId`; the shim assigns them
and returns the populated Task in the response.

### Routing metadata (NATS headers, not Message body)

Sesh-internal routing context — `sesh.machine`, `sesh.project`,
`sesh.session`, `sesh.role`, `sesh.inst` — travels as NATS message
**headers** when the shim publishes a prompt:

```
Sesh-Machine:   cc
Sesh-Project:   sesh
Sesh-Session:   rc-dev
Sesh-Role:      reviewer
Sesh-Inst:      abc123
Sesh-Principal: user:dan       # for sesh-ops Task.CreatedBy
```

A persistent copy of this routing context lives on the corresponding
**Task record** (in `Task.Metadata`), so mesh-internal observers
querying the Tasks bucket can filter by role/machine/etc. without
touching Messages. The Messages bucket stores A2A-pristine Messages.

### v2 subject

```
agents.prompt.v2.<machine>.<project>.<session>.<role>[.<inst>]
```

Seven or eight tokens. Adapters that support v2 subscribe to both this
prefix and the v1 `agents.prompt.*` prefix. Adapters that don't see no
v2 traffic at all — natural queue-group separation, no nack/re-route
dance.

The shim publishes v2 prompts on this subject. Internal CLI callers
that don't care about the v2 shape continue using v1 subjects.

### Reply chunk shape (streaming)

When a v2 prompt is being processed, the adapter publishes streaming
chunks on `agents.task.stream.<scope-kind>.<scope-id>.<task-id>` (core
NATS, ephemeral). Each chunk is one of the A2A StreamResponse union
shapes:

```json
{"task":          {...}}
{"message":       {...}}
{"statusUpdate":  {...}}
{"artifactUpdate":{...}}
```

The shim subscribes to this subject on behalf of any active SSE
clients, translating each NATS message 1:1 into an A2A SSE event.

### Final persisted state

When the task reaches a terminal state, the final Task object lives in
`sesh_tasks_<scope-kind>_<scope-id>`, all Messages in
`sesh_messages_<scope-kind>_<scope-id>`, and all completed Artifacts in
`sesh_artifacts_<scope-kind>_<scope-id>`. Late-arriving A2A clients use
`SubscribeToTask` to backfill from these durable stores plus tail any
remaining live chunks.

## Section 3 — NATS verb hierarchy

The full v0.4 subject map. v0.4 is **minimally additive** — three new
NATS subject categories on top of the existing v0.3 surface. Task CRUD,
Messages CRUD, Artifacts CRUD, and notification config CRUD all happen
via **direct JetStream KV access** (shim and adapters use sesh-ops as
a library); they do NOT have corresponding NATS RPC subjects.

### Adapter-served (NATS micro request/reply)

| Subject | Tokens | Status | Purpose |
|---|---|---|---|
| `agents.prompt.<agent>.<owner>.<name>` | 5 | existing | v1 direct-addressed prompt |
| `agents.prompt.<machine>.<project>.<session>.<role>[.<inst>]` | 6–7 | existing | v1 coord-routed prompt |
| `agents.prompt.v2.<machine>.<project>.<session>.<role>[.<inst>]` | 7–8 | **NEW** | v2 coord-routed prompt (envelope v2) |
| `agents.card.get.<agent>.<owner>.<name>` | 5 | **NEW** | L3 AgentCard contribution (public) |
| `agents.card.extended.<agent>.<owner>.<name>` | 5 | **NEW** | L3 AgentCard extended (auth-gated at shim) |
| `$SRV.INFO.agents` / `$SRV.PING.agents` / `$SRV.STATS.agents` | 3 | existing | NATS micro built-ins |

### Adapter-published (core NATS pub/sub)

| Subject | Tokens | Status | Purpose |
|---|---|---|---|
| `agents.hb.<agent>.<owner>.<name>` | 5 | existing | heartbeat |
| `agents.status.<agent>.<owner>.<name>` | 5 | existing | status |
| `agents.attachments.<agent>.<owner>.<name>` | 5 | existing | attachment notices |
| `agents.report.<machine>.<project>.<session>[.<role>[.<inst>]]` | 5–7 | existing | coord report |
| `agents.task.stream.<scope-kind>.<scope-id>.<task-id>` | 6 | **NEW** | live chunked output for active task (ephemeral; durable copy lands in KV) |

### What is **not** a NATS subject in v0.4 (deliberately)

The following data flows happen via **direct JetStream KV access** by
the shim and adapters (both import sesh-ops as a library); they do not
have intermediate NATS RPC subjects:

- **Task CRUD** — `task.Add/Get/List/Cancel/Subscribe` are sesh-ops
  library calls writing/reading `sesh_tasks_*` buckets directly.
- **Messages CRUD** — `messages.Append/History/Watch` operate on
  `sesh_messages_*` buckets directly.
- **Artifacts CRUD** — `artifacts.Put/Get/Watch` operate on
  `sesh_artifacts_*` buckets directly.
- **Notification config CRUD** — shim writes `sesh_notifycfg_*`
  buckets directly when handling A2A `CreateTaskPushNotificationConfig`
  etc.

NATS RPC subjects exist only where the receiver has behavior beyond
"persist to KV" — prompts (the receiver does work), cards (the
receiver advertises capabilities), heartbeats/status (the receiver
publishes liveness). Adding NATS RPC tiers in front of KV writes would
be pass-through abstraction with no behavior to hide.

**Out of v0.4 scope** (deferred to separate designs):
- `agents.event.<type>.*` — sesh-wide observability lifecycle events
  (session-start, tool-use, etc.). Not part of A2A parity; will be
  designed separately if/when a consumer requires them.
- `agents.shutdown.*` — graceful adapter shutdown. Not part of A2A
  parity; underspecified. Sesh already detects adapter loss via
  heartbeat absence.

### NATS subject design rules validated

The verb hierarchy was validated against the `nats-design-subject`
skill's criteria:

- ✓ **Broad-to-specific ordering** — domain (`agents`) → action (verb)
  → identity or scope tokens → task-id
- ✓ **Low-cardinality left, high-cardinality right** — task-id and
  instance-id always rightmost
- ✓ **Wildcard subscription friendliness** —
  `agents.task.stream.session.foo.>`, `agents.prompt.v2.>`,
  `agents.card.get.>` all work as natural observer filters
- ✓ **Naming convention** — lowercase, hyphens not underscores, ULIDs
  for IDs
- ✓ **Subject permissions** — `agents.>` is the natural NATS account
  boundary; per-scope can be added via
  `agents.task.stream.session.foo.>` if needed
- ⚠ **6-segment soft limit exceeded** by
  `agents.prompt.v2.<5-coord-tokens>` (7–8 tokens). Accepted because:
  - The skill's 4–6 segment guideline targets JetStream stream-filter
    complexity. Our prompt subject is core NATS pub/sub — no
    JetStream stream involvement.
  - The token-position filtering benefit is real: ops can subscribe to
    `agents.prompt.v2.>.>.>.implementer.>` for "all v2 prompts going
    to implementer agents across the mesh".
  - Precedent exists — the v0.3 coord subject already runs 7 tokens.

Note: dropping the `agents.task.*`, `agents.event.*`,
`agents.shutdown.*`, and `agents.notify.*` subject categories from
earlier drafts reduced the long-subject count significantly. The
remaining v0.4 subjects mostly fit the 5–6 token recommendation; only
the v2 prompt subject exceeds it (consistent with the existing v1
coord subject).

### Scope-id sanitization

NATS bucket names and subject tokens allow alphanumeric + `-` + `_`.
Scope-id values from the wild may contain `.`, `:`, `/`, space.
Sanitization rule (applied identically by sesh-ops and shim):

1. Replace every char in `[^a-zA-Z0-9_-]` with `-`
2. Truncate to 64 chars
3. If step 1 or 2 changed the value, append `_<8-char-hash-of-original>`
   to preserve uniqueness

Example: `session:rc-dev/2026-05-24` → `session-rc-dev-2026-05-24` (no
hash needed since truncation didn't fire and the original is still
recoverable by reverse-mapping the hyphens).

## Section 4 — Companion buckets schema

### Bucket layout

Three sibling KV buckets per scope instance plus one shared Object Store:

| Bucket | Status | A2A surface backed |
|---|---|---|
| `sesh_tasks_<scope-kind>_<scope-id>` | existing (sesh-ops) | Task |
| `sesh_messages_<scope-kind>_<scope-id>` | **NEW** | Message (full history) |
| `sesh_artifacts_<scope-kind>_<scope-id>` | **NEW** | Artifact (persisted, post-stream assembly) |
| `sesh_objects_<scope-kind>_<scope-id>` (Object Store) | **NEW** | File parts >1MB |
| `sesh_notifycfg_<scope-kind>_<scope-id>` | **NEW** | A2A push notification configs |
| `sesh_notifyfail_<scope-kind>_<scope-id>` | **NEW** | dead-lettered push notification deliveries |

**AgentCard caching note**: there is no `sesh_cards_*` bucket. AgentCard
composition runs at fetch time via `agents.card.get.*` request/reply
and is cached **in-process** in the shim (5-minute TTL). Each shim
replica maintains its own LRU. A KV-backed shared cache was considered
and rejected — two cache layers double the invalidation surface
without a clear scaling win at sesh's expected agent count.

**Naming consistency note**: sesh-ops's existing bucket convention is
`sesh_<kind>_<scope>_<scope-id>` where `<kind>` is one of
`tasks|events|blobs|memory` and is derived via
`scope.Bucket(scope, id, kind)`. v0.4 extends `<kind>` with
`messages`, `artifacts`, `objects`, `notifycfg`, `notifyfail`.

### Key shapes

| Bucket | Key | Watch pattern |
|---|---|---|
| Tasks | `<task-id>` | `<task-id>` (one task) or `>` (all in scope) |
| Messages | `<task-id>.<message-id>` | `<task-id>.>` (all messages for one task) or `>` (all in scope) |
| Artifacts | `<task-id>.<artifact-id>` | `<task-id>.>` (all artifacts for one task) |
| Object Store | `<task-id>/<artifact-id>` | — (Object Store; no KV watch) |
| NotifyConfigs | `<task-id>.<config-id>` | `<task-id>.>` (all configs for one task) |
| NotifyFailed | `<task-id>.<config-id>.<attempt>` | `<task-id>.>` (ops inspection) |

**Object Store key note**: artifact chunks travel ephemerally over
core NATS (`agents.task.stream.*`); only the **assembled** artifact
lands in Object Store. Key is `<task-id>/<artifact-id>` — no
chunk-index segment. Chunked-resumable uploads are not part of v0.4;
if needed later, key shape grows to include a part index.

Both task-id and message-id are ULIDs → keys are time-sortable.

### Value schemas (byte-for-byte A2A)

The whole point of mirroring A2A's shape in storage is that the shim
passes values through with zero translation cost.

**Message value** (byte-for-byte A2A `Message` per
`a2a-go/a2a/core.go:199`):

```json
{
  "messageId": "01HX8K2N7QGZ9Y3VRWMTBV5DPE",
  "taskId": "01ARYZ6S41TSRCOSKF2FVG69G2",
  "contextId": "session:rc-dev",
  "role": "user",
  "parts": [
    {"text": "...", "mediaType": "text/plain"},
    {"url":  "obj://sesh_objects_session_rc-dev/<task>/<art>",
     "mediaType": "application/pdf",
     "filename": "doc.pdf"},
    {"data": {"foo": "bar"}, "mediaType": "application/json"}
  ],
  "extensions": ["https://example.com/ext/v1"],
  "referenceTaskIds": [],
  "metadata": {}
}
```

The value is **A2A-pristine** — no `sesh.*` fields. Sesh routing
context (machine/role/inst/etc.) lives in NATS message headers during
transit and in the corresponding **Task** record's `Metadata` field for
durable query. Mesh-internal observers filter by role/machine via Task
queries, not Messages queries.

**Artifact value** (per `a2a-go/a2a/core.go:418`):

```json
{
  "artifactId": "01HX8K3M2J4PQRSTU7VWXYZA8B",
  "name": "summary.md",
  "description": "post-review summary",
  "parts": [
    {"text": "## Findings\n\n...", "mediaType": "text/markdown"}
  ],
  "extensions": [],
  "metadata": {}
}
```

(`extensions` is part of the A2A Artifact shape — array of extension
URIs relevant to this artifact.)

Streaming chunks flow over `agents.task.stream.*` as
`TaskArtifactUpdateEvent` shape (`{artifact, append, lastChunk}`).
Adapters assemble locally; when `lastChunk: true`, write the final
Artifact to KV.

### CAS semantics

- **Messages**: append-only. Writer specifies `Create` (CAS revision =
  0). Duplicate `messageId` → conflict, retry with new ID (or treat as
  idempotent success — see Section 9).
- **Artifacts**: `Update` with last-known revision allowed (chunk
  reassembly may produce updates). Concurrent writers serialize via
  CAS.
- **Tasks**: existing sesh-ops CAS unchanged.

### Retention and TTL

| Bucket | TTL policy | History depth |
|---|---|---|
| Tasks | per sesh-ops policy (typically MaxAge per bucket) | 1 (no history needed; state is current) |
| Messages | same as parent Task bucket | **10** (supports last-N-message replay without full scan) |
| Artifacts | same as parent Task bucket | 1 (only latest assembled version matters) |
| Object Store | independent JetStream Object Store TTL; KV pointer-only references auto-orphan on blob delete | n/a |
| Cards | 7 days, refreshed on adapter restart | 1 |
| NotifyConfigs | same as parent Task bucket | 1 |
| NotifyFailed | 7 days (ops inspection window) | 1 |

A background sweeper in sesh-ops deletes Messages and Artifacts whose
Task ID no longer exists in the Tasks bucket (orphan cleanup, runs
every 6 hours).

### Object Store thresholds

- Inline in Message/Artifact Part: ≤ **1 MB** base64-encoded (A2A
  `bytes` field)
- Off-load to Object Store: > 1 MB → `uri` field with `obj://` scheme

The adapter SDK helper handles the threshold check transparently. The
shim translates `obj://` URIs to HTTPS download URLs at the gateway
boundary.

### Bucket-explosion mitigation

Session-scoped buckets multiply: one Tasks + Messages + Artifacts
triple per active session. At 100 active sessions, 300 KV buckets +
~100 Object Stores. NATS handles thousands of KV buckets fine but
stream count grows linearly.

Mitigation — sesh-ops grows a **bucket pruning policy**: delete a
bucket triple when its Tasks bucket has been empty for >7 days.
Implemented as a periodic job in sesh-ops; configurable.

### Re-attach / replay (A2A `SubscribeToTask`)

Two-layer mechanism:

1. **Live tail**: subscribe `agents.task.stream.<scope>.<scope-id>.<task-id>`
   (core NATS, ephemeral). Don't miss in-flight chunks.
2. **Backfill**: KV watch on `sesh_messages_<scope>_<scope-id>` with
   key prefix `<task-id>.>` and `IncludeHistory: true` — replays all
   messages from KV history, then transitions to live updates as new
   messages are written.

The shim composes these: opens live tail first, then runs the KV
backfill, then de-dupes by `messageId`. Standard "join the live stream,
then catch up" pattern.

## Section 5 — State machine mapping

### A2A TaskState (9 values, wire format)

Per `a2a-go/a2a/core.go:279-301`. **Wire format is SCREAMING_SNAKE_CASE**:

| State (wire) | Terminal? | Meaning |
|---|---|---|
| `TASK_STATE_SUBMITTED` | no | accepted, not yet started |
| `TASK_STATE_WORKING` | no | actively executing |
| `TASK_STATE_INPUT_REQUIRED` | no | paused awaiting user/system input |
| `TASK_STATE_AUTH_REQUIRED` | no | paused awaiting auth/credentials |
| `TASK_STATE_COMPLETED` | **yes** | successful |
| `TASK_STATE_CANCELED` | **yes** | aborted by user/system |
| `TASK_STATE_FAILED` | **yes** | errored |
| `TASK_STATE_REJECTED` | **yes** | agent refused to process |
| `TASK_STATE_UNSPECIFIED` | no | wire-only fallback when state cannot be determined |

The SDK exposes `TaskState.Terminal()` (core.go:330-337) which
returns true for the four terminal states. A2A clients use this rather
than a separate `final` boolean on stream events (see Section 5's
"Wire format for state updates" subsection).

### sesh-ops Status (6 values, current)

`pending | in_progress | blocked | completed | failed | cancelled`

These remain the **internal** Go enum values in sesh-ops (lowercase
underscore-style, matching existing convention). The shim translates
to/from SCREAMING_SNAKE_CASE at the wire boundary.

### Mapping decision: targeted extension

Of three options considered (wire-only A2A; grow sesh-ops to full 9
states; targeted extension), this document chooses **targeted
extension**:

- Add `StatusRejected` to sesh-ops Status enum (1 new value)
- Add `BlockedReason` discriminator field to Task (optional, defaults empty)
- `TASK_STATE_UNSPECIFIED` is wire-only — never persisted in sesh-ops

| A2A TaskState (wire) | sesh-ops Status (Go) | BlockedReason |
|---|---|---|
| `TASK_STATE_SUBMITTED` | `pending` | — |
| `TASK_STATE_WORKING` | `in_progress` | — |
| `TASK_STATE_INPUT_REQUIRED` | `blocked` | `input-required` |
| `TASK_STATE_AUTH_REQUIRED` | `blocked` | `auth-required` |
| `TASK_STATE_COMPLETED` | `completed` | — |
| `TASK_STATE_CANCELED` | `cancelled` | — |
| `TASK_STATE_FAILED` | `failed` | — |
| `TASK_STATE_REJECTED` | `rejected` *(NEW)* | — |
| `TASK_STATE_UNSPECIFIED` | *(never persisted; shim emits only on genuine resolution failure, prefers `TaskNotFoundError` instead)* | — |

`BlockedReason` also lights up an existing-but-implicit sesh-ops
behavior — `BlockedReasonDependency` surfaces "task is parked waiting
on a `DependsOn` graph parent" as a first-class state.

### sesh-ops schema delta

```go
const (
    // ... existing 6 ...
    StatusRejected Status = "rejected"  // NEW
)

type BlockedReason string
const (
    BlockedReasonNone          BlockedReason = ""
    BlockedReasonInputRequired BlockedReason = "input-required"
    BlockedReasonAuthRequired  BlockedReason = "auth-required"
    BlockedReasonDependency    BlockedReason = "dependency"
)

type Task struct {
    // ... existing fields ...
    BlockedReason BlockedReason `json:"blocked_reason,omitempty"`
}
```

### Legal transitions

```
                  ┌─────────────┐
                  │   pending   │ (A2A: submitted)
                  └──────┬──────┘
                         │ Lease / start
                         ▼
                  ┌─────────────┐
              ┌──►│ in_progress │ (A2A: working)
              │   └──┬───┬───┬──┘
              │      │   │   │ Pause for input/auth/dep
              │      │   │   ▼
              │      │   │  ┌─────────┐
              │      │   │  │ blocked │ (+BlockedReason)
              │      │   │  └────┬────┘
              │      │   │       │ Unblock
              └──────┼───┼───────┘
                     │   │
                     │   │ Cancel (user/system)
                     │   ▼
                     │  ┌──────────┐
                     │  │cancelled │ (A2A: canceled)
                     │  └──────────┘  [terminal]
                     │
                     │ Complete / Fail / Reject
                     ▼
                  ┌─────────┬────────┬──────────┐
                  │completed│ failed │ rejected │
                  └─────────┴────────┴──────────┘  [terminal]
```

**Forbidden transitions** (enforced via CAS-protected status updates in
sesh-ops):

- Any terminal → anything (terminal is sticky)
- `pending` → terminal (must go through `in_progress` or `cancelled`)
- `blocked` → `pending` (use `in_progress`; `pending` is pre-lease only)

Sesh-ops rejects these transitions at the API layer with a typed
`ErrInvalidTransition` so the shim can map it to A2A `-32602 Invalid
params`.

### Atomic status+message API

Sesh-ops grows a new method:

```go
func (s *Service) UpdateStatusWithMessage(
    ctx context.Context,
    taskID string,
    newStatus Status,
    reason BlockedReason,
    explanationMessage *MessageRef,
) error
```

This writes the Task status change AND (if `explanationMessage` is
non-nil) writes a Message to the Messages bucket in a single CAS
round. The shim populates `TaskStatusUpdateEvent.status.message` from
this most-recent-explanation Message.

### Wire format for state updates

When sesh-ops writes a Task status change, the KV watcher fires; the
shim translates each delta into A2A `TaskStatusUpdateEvent` (per
`a2a-go/a2a/core.go:516`):

```json
{
  "taskId": "01ARYZ6S41TSRCOSKF2FVG69G2",
  "contextId": "session:rc-dev",
  "status": {
    "state": "TASK_STATE_INPUT_REQUIRED",
    "message": {
      "messageId": "01HX...",
      "role": "agent",
      "parts": [{"text": "Need approval to delete files. Reply Y/N.",
                 "mediaType": "text/plain"}]
    },
    "timestamp": "2026-05-24T18:30:00Z"
  }
}
```

**Note on absent `final` field**: earlier A2A drafts had a `final`
boolean on `TaskStatusUpdateEvent`. **It was removed in v1.0.0-alpha**
(per a2a-go CHANGELOG, 2026-02-23) — clients now determine finality
by calling `TaskState.Terminal()`. Our SSE bridge stops emitting after
a terminal state and closes the stream; no separate flag.

### Cancellation semantics

A2A `CancelTask` can be called from any non-terminal state.
Sesh-ops's existing `CancelTask` moves any of
`pending|in_progress|blocked` → `cancelled`. Shim translates 1:1.

Edge case: A2A spec says rejecting cancel of a terminal task returns
`-32002 TaskNotCancelable`. Sesh-ops's existing `CancelTask` returns
nil on idempotent re-cancel of already-cancelled. To disambiguate, add
a typed `ErrTaskTerminal` distinct from `ErrTaskNotFound` and the
idempotent-already-cancelled path.

## Section 6 — AgentCard schema

### Three-layer composition

Cards merge bottom-up across three sources:

| Layer | Source | Contributes |
|---|---|---|
| **L1 Sesh defaults** | Shim config (`shim.toml`) | `protocolVersion`, `supportedInterfaces[]` (HTTPS gateway URL + JSON-RPC binding), `provider`, `securitySchemes`, `capabilities.streaming/pushNotifications/extendedAgentCard` |
| **L2 Adapter metadata** | `$SRV.INFO.<service>` metadata | `name`, `version`, `defaultInputModes`, `defaultOutputModes`, agent identity |
| **L3 Adapter-served card (public)** | `agents.card.get.<agent>.<owner>.<name>` | `description`, `skills[]`, `iconUrl`, `documentation`, `capabilities.extensions[]` |
| **L3 Adapter-served card (extended)** | `agents.card.extended.<agent>.<owner>.<name>` | Same shape as L3 public + additional skills/fields exposed only to auth-passing clients. Adapters that don't serve this subject → shim returns `AuthenticatedExtendedCardNotConfiguredError`. |

Higher layer overrides lower. Shim runs the merge at card-fetch time
with caching.

### L1 — sesh defaults

Carries values that don't vary per agent: HTTPS gateway URL,
authentication scheme, that streaming is always supported, the
operator's organization. Lives in `shim.toml`.

### L2 — `$SRV.INFO.agents`

Reuses the existing NATS micro service-info channel. Metadata block on
the existing handler carries:

```
sesh.protocol_version    = "0.3" | "0.4"
sesh.v04_capabilities    = "messages,artifacts,cards,events,shutdown"
agent, owner, name, role, class, machine, project_id
```

For v0.3 adapters lacking the v0.4 metadata keys, the shim composes a
skeleton card from L1+L2 — still well-formed, just missing the rich
fields.

### L3 — `agents.card.get.*` (public) and `agents.card.extended.*` (auth-gated)

Adapters that want a rich card register a NATS micro handler for
`agents.card.get.<agent>.<owner>.<name>`. The reply is a partial
AgentCard contribution containing the fields adapters genuinely vary
on:

```
SUBJECT:  agents.card.get.<agent>.<owner>.<name>
REQUEST:  {}  (identity is in the subject)
REPLY:    {
            "description": "Code review agent specialized in Go and Rust",
            "skills": [
              {
                "id": "review-pr",
                "name": "Review pull request",
                "description": "Reviews a GitHub PR diff and posts findings",
                "tags": ["github", "code-review"],
                "examples": ["review PR #123 on owner/repo"],
                "inputModes": ["text/plain"],
                "outputModes": ["text/markdown", "application/json"]
              }
            ],
            "iconUrl": "https://...",
            "documentation": "https://...",
            "capabilities": {
              "extensions": [
                {"uri": "https://example.com/ext/code-review/v1",
                 "required": false}
              ]
            }
          }
```

Adapters that want to expose **additional skills only to
authenticated A2A clients** register a *second* handler on
`agents.card.extended.<agent>.<owner>.<name>`. The reply has the same
shape as the public one but typically includes more skills. The shim
authenticates at the HTTPS boundary, and only calls the extended
subject when the principal carries the required auth scope (see
Section 7 — Auth differentiation).

If an adapter doesn't serve the extended subject, the shim returns
`AuthenticatedExtendedCardNotConfiguredError` to the client. This is
the cleanest auth boundary — no inline filter contract leaking from
the shim into adapter response shapes.

### Skill ID convention

`<agent-name>.<verb>-<noun>` lowercase-hyphenated.
`description` is the human-facing summary; `examples` populate A2A's
example prompts for client UX hints. `inputModes`/`outputModes`
per-skill override adapter defaults.

### Security schemes (L1-owned)

The shim is the HTTPS boundary, so it owns security advertising.
Default scheme:

```json
{
  "bearer": {
    "type": "http",
    "scheme": "bearer",
    "bearerFormat": "JWT",
    "description": "Bearer token issued by sesh-auth or upstream IdP"
  }
}
```

Operator can override in `shim.toml` to add OAuth2, OpenIdConnect, or
ApiKey schemes. Adapters never see auth tokens — the shim authenticates
at HTTPS, then passes a sesh-internal identity hint as NATS message
metadata.

### Signing (JCS + JWS)

Per A2A spec, AgentCards SHOULD be signed:

1. Compose merged card (L1+L2+L3)
2. JCS-canonicalize (RFC 8785)
3. JWS-sign with shim's private key (ES256 default)
4. Attach `signatures: [{protected, signature}]` to card
5. Cache signed card

Key management: shim holds a long-lived private key in config or
KMS/Vault. Recommendation: ES256 with key rotation via JWKS endpoint at
`/.well-known/jwks.json`. Keys live in `~/.config/sesh/shim/keys/` for
local dev, KMS/Vault in prod.

### Caching

| Cache layer | TTL | Invalidation |
|---|---|---|
| Per-agent merged card (in-process LRU) | 5 min | TTL expiry; force refresh on `$SRV.PING` failure for the source adapter |
| Signed bytes | same as merged card | Re-sign on card change |

Each shim replica caches independently. A shared KV-backed cache was
considered and rejected — two cache layers double the invalidation
surface without a clear scaling win.

### Discovery within the mesh

Internal sesh tooling shouldn't go through HTTPS to discover agents:

- `$SRV.INFO.agents` — quick discovery, L2 only, no card composition
  cost
- `agents.card.get.>` — for tools that want the full composed card,
  called over NATS directly
- `sesh mesh card <agent>` (NEW subcommand) — returns the composed
  AgentCard for inspection, useful for debugging shim output without
  going through the gateway

### SDK helper

Sesh-channels TS SDK gets a new helper:

```typescript
import { registerAgentCard } from '@agent-ops/sesh-channels';

registerAgentCard({
  description: "Code review agent",
  skills: [
    {
      id: "review-pr",
      name: "Review pull request",
      // ...
    }
  ],
  iconUrl: "https://...",
});
```

Internally: registers the NATS micro handler for `agents.card.get.*`.
A second helper, `registerAgentCardExtended(...)`, registers the
auth-gated `agents.card.extended.*` handler if the adapter wants to
expose additional skills only to authenticated A2A clients. Adapter
authors don't deal with NATS subjects directly.

## Section 7 — Shim architecture

### Deployment topology

The shim is a single Go binary at `sesh/cmd/sesh-shim/`. Stateless
except for its signing key — scale-out is N replicas behind a load
balancer sharing the same KV-backed state. No sticky sessions for
non-streaming requests; SSE streams ARE sticky to the originating shim
for their duration.

Default deployment: one shim per NATS server (or one per cluster, with
LB). Alternative: per-scope shims for blast-radius limiting in shared
infra.

### Process model

Single Go binary, not embedded in adapters. Reasoning:
- Adapters are TS; embedding a Go HTTPS server in Node is awkward
- Auth/signing/HTTPS concerns are orthogonal to adapter concerns
- One binary to operate matches sesh's existing CLI/daemon pattern

### Responsibilities

1. Protocol translation — A2A JSON-RPC ↔ NATS pub + JetStream KV
   reads/writes (via sesh-ops library)
2. Auth — HTTPS-layer authentication, map to sesh-internal principal
3. Card composition + signing — L1+L2+L3 merge, JCS canonicalize, JWS sign, cache
4. SSE bridge — KV watch + core NATS `agents.task.stream.*` →
   `TaskStatusUpdateEvent`/`TaskArtifactUpdateEvent` SSE chunks
5. Push notifications — webhook delivery, retry, backoff, dead-letter
6. Error mapping — sesh-ops library errors + NATS errors → A2A JSON-RPC errors
7. Object Store URI translation — `obj://` ↔ HTTPS download URLs
8. Rate limiting — per-principal, configurable
9. Subject-builder: typed subject construction via shared package
   (also used by adapters)

**Not** shim responsibilities: Task state-machine rules (enforced
inside sesh-ops library at write time), Message persistence (Messages
bucket, written by adapter SDK during processing), AgentCard skill
content (adapter L3).

### A2A method surface implemented

The shim implements the **JSON-RPC binding** of the A2A protocol.
Method names use the PascalCase form from `a2a-go/a2asrv/handler.go`:

| Method | Purpose | Backend operation |
|---|---|---|
| `SendMessage` | submit a Message; create-or-append Task | KV write + `agents.prompt.v2.*` publish |
| `SendStreamingMessage` | submit + open SSE for live updates | same + KV watch + `agents.task.stream.*` subscribe |
| `GetTask` | fetch Task by id (with optional `historyLength`) | KV read |
| `ListTasks` | enumerate tasks in principal-authorized scopes (filters: status, contextId, date) | KV scan + post-filter |
| `CancelTask` | cancel a non-terminal task | sesh-ops library: `task.Cancel(...)` → KV write |
| `SubscribeToTask` | reattach to an in-flight task (live + history) | KV watch with `IncludeHistory: true` + `agents.task.stream.*` subscribe |
| `CreateTaskPushNotificationConfig` | register webhook for status changes | KV write to `sesh_notifycfg_*` |
| `GetTaskPushNotificationConfig` | fetch webhook config | KV read |
| `ListTaskPushNotificationConfigs` | list webhook configs for a task | KV scan |
| `DeleteTaskPushNotificationConfig` | remove webhook config | KV delete |
| `GetExtendedAgentCard` | fetch the auth-gated AgentCard | NATS request to `agents.card.extended.*` + L1+L2 merge |

### Request lifecycle: `SendMessage`

```
1. HTTPS POST /a2a
   Authorization: Bearer <jwt>
   {"jsonrpc":"2.0","id":1,"method":"SendMessage",
    "params":{"message":{...},"configuration":{"returnImmediately":false}}}

   Note: configuration.returnImmediately defaults to false (block until
   task terminal). Set to true for fire-and-forget semantics.

2. SHIM: validate JWT, extract principal
   principal = {sub: "user:dan", scopes: [...]}

3. SHIM: resolve target agent
   - if message.contextId populated → look up existing Task's scope
   - else → derive from routing hint (header or shim default routing)
   yields: (scope-kind, scope-id, agent-id)

4. SHIM: ensure Task exists (via sesh-ops library)
   if message.taskId not set:
       task = task.Add(ctx, kv, AddOpts{CreatedBy: principal, ...})
       message.taskId = task.ID
   else:
       task = task.Get(ctx, kv, message.taskId)
       authorize principal against task.CreatedBy/Metadata

5. SHIM: write Message to KV (via sesh-ops library)
   messages.Append(ctx, kv, AppendOpts{Task: task.ID, Msg: message})
   internal: CAS Create on key <task-id>.<message-id>

6. SHIM: publish prompt subject (core NATS)
   subj:    agents.prompt.v2.<machine>.<project>.<session>.<role>[.<inst>]
   headers: Sesh-Machine/Project/Session/Role/Inst/Principal
   body:    {messageId, taskId, contextId}
   (queue-subscribed by one v0.4-capable adapter in the role pool)

7. ADAPTER (sesh-channels): receives prompt, processes
   uses sesh-ops library to update task status, append agent Messages,
   write completed Artifacts; publishes streaming chunks on
   agents.task.stream.<scope-kind>.<scope-id>.<task-id>

8. SHIM:
   if returnImmediately=false (default):
       KV watch sesh_tasks_<scope>_<scope-id> key=<task-id>
       wait until task.Status.Terminal()
       read final Task + Artifacts via sesh-ops library
       return JSON-RPC response with Task object
   if returnImmediately=true:
       return Task object with current status immediately
```

### Streaming variant: `SendStreamingMessage`

Same as above through step 6. Step 7+ becomes SSE:

```
7'. SHIM: open SSE response (content-type text/event-stream)
   core NATS subscribe: agents.task.stream.<scope>.<scope-id>.<task-id> (live)
   KV watch: sesh_tasks_<scope>_<scope-id>      key=<task-id>
   KV watch: sesh_messages_<scope>_<scope-id>   prefix=<task-id>.>
   KV watch: sesh_artifacts_<scope>_<scope-id>  prefix=<task-id>.>

8'. SHIM: for each NATS message OR KV event:
   translate to A2A event envelope ({task|message|statusUpdate|artifactUpdate})
   emit as SSE data: line
   after emitting a statusUpdate with state.Terminal()==true: close stream

9'. SHIM: on client disconnect, cancel subscriptions
   (task continues running mesh-side; client can SubscribeToTask to reattach)
```

### Reattach: `SubscribeToTask`

A2A's reattach primitive — the same RPC used for in-flight late-joiners
and post-disconnect resumption:

1. Validate the reattaching principal has access to taskId (via Task
   record's CreatedBy and authorization rules)
2. Open new SSE
3. KV watch with `IncludeHistory: true` on Messages + Artifacts
   prefix=`<task-id>.>` → backfills missed events
4. Start core NATS subscription on `agents.task.stream.*` → live tail
5. De-dupe in-flight by (messageId, artifactId)

### Push notifications

A2A clients configure webhooks via
`CreateTaskPushNotificationConfig`. Shim stores configs in
`sesh_notifycfg_<scope>_<scope-id>` keyed by `<task-id>.<config-id>`.

For **at-least-once delivery**, the worker uses a JetStream durable
consumer on the Tasks KV stream (ack semantics, retry-on-nack), not a
naive in-memory KV watch:

```
JetStream durable consumer on Tasks bucket
  → on status change event:
      look up matching notify configs for this taskId
      POST to webhook with A2A TaskStatusUpdateEvent body
      ack on 2xx response; nack with backoff on failure
      retry with exponential backoff (1s, 4s, 16s, 64s)
      after exhaustion → write entry to sesh_notifyfail bucket
```

Dead-lettered notification attempts go to
`sesh_notifyfail_<scope>_<scope-id>` for ops inspection (7-day TTL).
This is *attempts*, not configs — see Section 4. Webhook receivers
must be idempotent (A2A clients are expected to handle duplicate
deliveries per spec).

**SSRF protection** (security): shim validates webhook URLs on
registration AND at delivery time:
- Reject schemes other than `https://` (operator-configurable to allow
  `http://localhost`/`127.0.0.1`/`::1` for dev)
- Reject hostnames resolving to private IP ranges (RFC 1918,
  169.254/16, ULA fc00::/7, loopback) — resolve at both registration
  and delivery to defeat DNS rebinding
- Optional operator-configured allowlist of webhook hostnames

**Credential storage**: `PushConfig.Auth.Credentials` is encrypted at
rest using a shim-held symmetric key (or KMS-managed key). KV stores
`enc:<base64>`; plaintext only at delivery time.

### Failure modes

| Failure | Shim behavior |
|---|---|
| NATS unreachable | 503 to all HTTPS requests; reconnect with backoff |
| Adapter not present for subject | Per-request: `-32601`. Card composition: L1+L2 skeleton |
| KV bucket missing | Auto-create on first write (sesh-ops API); retry once after delay |
| Signing key file missing | Fail-fast at startup; AgentCard endpoints unavailable |
| Auth provider down | 503 on auth-required requests; cached AgentCard still served |
| SSE client disconnects mid-stream | Cancel NATS subscriptions; task continues mesh-side |
| Stream chunk lost (core NATS ephemeral) | KV history backfill catches durable state on resubscribe |

### Health and observability

- `GET /healthz` — process up, NATS connected, signing key loaded
- `GET /readyz` — health + at least one adapter responding to
  `$SRV.PING.agents`
- Prometheus `/metrics` — per-method counters/latencies, NATS RTTs,
  cache hit rates, push delivery rates
- Structured JSON logs to stdout

### Explicitly out of scope

- Service mesh / Envoy sidecar features (operators add their own)
- A2A payload schema validation beyond what's needed to route
- Multi-protocol gateway (gRPC, GraphQL) — A2A is HTTPS+JSON-RPC

## Section 8 — Migration path

The whole design is additive. Every existing sesh deployment continues
to work with zero changes. v0.4 capabilities ride on top, opt-in per
adapter, per scope, per shim. No flag day.

### Compatibility guarantees

1. All existing v0.3 subjects keep their semantics
2. Existing sesh-ops Task consumers see no behavior change
3. Adapters without L3 AgentCard still appear in the mesh (skeleton
   card from L1+L2)
4. A mesh with no shim deployed continues to function for sesh-native
   tooling

### Capability discovery

Adapters advertise v0.4 participation via `$SRV.INFO` metadata:

```
metadata.sesh.protocol_version  = "0.3" | "0.4"
metadata.sesh.v04_capabilities  = "messages,artifacts,cards"
                                  (comma-separated; absent means none)
```

Shim's per-agent code path:
- If `protocol_version >= 0.4` and capability advertised → use new path
- Else → fall back (skeleton AgentCard, no v2 subjects published)

### Rollout sequence (five phases)

Each phase ships independently; later phases assume earlier ones merged.

#### Phase 1 — sesh-ops: status + reason extensions

- Add `StatusRejected` const
- Add `BlockedReason` type + 4 const values
- Add `BlockedReason` field to `Task` struct (omitempty)
- Add `UpdateStatusWithMessage(...)` method (atomic Tasks+Messages CAS;
  Messages-write is no-op if Messages bucket doesn't exist yet)
- Add `ErrTaskTerminal` typed error

Migration: zero. Old Task JSON deserializes cleanly; new field defaults
to empty.

Verifies: existing sesh-ops tests pass; new tests for `rejected`
transition and `BlockedReason` round-trip.

#### Phase 2 — sesh-ops: companion bucket APIs

- New package `internal/messages/` with `Bucket.Append`, `History`, `Watch`
- New package `internal/artifacts/` with `Bucket.Put`, `Get`, `Watch`
- New package `internal/objects/` wrapping NATS Object Store
- All buckets lazy-created on first write
- Naming + sanitization per Section 4

Migration: zero. New buckets, no existing schemas touched.

Verifies: integration tests against a real NATS that create/read/watch
all three buckets.

#### Phase 3 — sesh: shim binary skeleton

- New cobra/kong command tree at `cmd/sesh-shim/`
- HTTPS+JSON-RPC server (listening + auth + routing only)
- AgentCard composer (L1+L2 only first; L3 added in Phase 5)
- JCS/JWS signer for cards
- `/.well-known/agent-card.json` endpoint
- Health/ready/metrics endpoints
- Imports sesh-ops as library; no RPC tier
- Implements TWO methods initially:
  `GetExtendedAgentCard` and `GetTask`

Migration: zero. New binary; nothing else changes.

Verifies: deploy to dev mesh, fetch AgentCard for an existing v0.3
adapter, confirm skeleton card is well-formed and JWS-verifies.

#### Phase 4 — sesh-channels SDK: envelope v2 + Messages writer

Split into 4a and 4b — each releases as its own SDK minor version so
adapters can adopt chat-only A2A parity (4a) without committing to
artifact streaming yet.

**Phase 4a — chat path:**
- Add envelope-v2 detection (subject prefix-based)
- Subscribe to `agents.prompt.v2.*` in addition to v1 subjects
- For v2 prompts: SDK writes the Message to `sesh_messages_*` on
  receipt; also writes adapter's reply Messages
- Add metadata advertisement: `metadata.sesh.protocol_version = "0.4"`
  and `metadata.sesh.v04_capabilities = "messages"`

**Phase 4b — artifacts + streaming + cards:**
- For v2 prompts: SDK writes Artifacts to `sesh_artifacts_*` on
  completion
- For v2 prompts: SDK publishes streaming chunks on
  `agents.task.stream.*`
- Add `registerAgentCard(...)` helper for L3 contribution (public)
- Add `registerAgentCardExtended(...)` helper for L3 contribution (auth-gated)
- Bump capability metadata to `"messages,artifacts,cards"`
- Bump SDK dep in all 5 adapters; default protocol_version stays "0.3"
  unless adapter opts in

Migration: zero. Adapters that don't register an AgentCard get a
skeleton card via shim L1+L2. v1 prompts continue unchanged.

Verifies: rolling upgrade one adapter at a time; legacy v0.3 tooling
still sees them via `sesh mesh ls`.

#### Phase 5 — sesh: shim full A2A surface

- Implement remaining A2A methods: `SendMessage`, `SendStreamingMessage`,
  `ListTasks`, `CancelTask`, `SubscribeToTask`,
  `CreateTaskPushNotificationConfig`,
  `GetTaskPushNotificationConfig`, `ListTaskPushNotificationConfigs`,
  `DeleteTaskPushNotificationConfig`, `GetExtendedAgentCard`
- SSE bridge (KV watch + core NATS subscribe)
- Push notification worker (JetStream durable consumer for
  at-least-once delivery; SSRF protection; credential encryption)
- L3 AgentCard merge with caching
- Object Store URI translation

Migration: zero. Shim is additive.

Verifies: end-to-end test from an external A2A client (e.g.,
`a2a-python`) against a v0.4-capable adapter.

### Per-scope opt-in

Operators can enable v0.4 features per-scope by:
1. Deploying the shim with `scope.scope_kind` + `scope.scope_ids`
   configured
2. Upgrading adapters in that scope's session(s) to SDK ≥ v0.4
3. Other scopes continue running v0.3 adapters with no shim

This lets ops dogfood v0.4 in one session before mesh-wide rollout.

### Adapter upgrade order (recommendation)

`pi` first (canary, simplest), then `claude` (highest-value), then
`omp`, `gemini`, `grok-nats-channel` in any order. Each upgrade is
independent.

| Phase | Rollback |
|---|---|
| 1 — sesh-ops schema | Any `rejected` tasks written during the window must be remapped to `failed` before downgrade; gate `rejected` behind a feature flag for the first release |
| 2 — companion buckets | Delete the Messages/Artifacts buckets; revert sesh-ops binary |

Lazy. Buckets created on first Message/Artifact write via NATS KV's
`CreateOrUpdate` bucket primitive. Avoids bucket explosion for

Message.

## Section 9 — Error handling

Errors cross three trust boundaries: HTTPS↔shim, shim↔NATS,
NATS↔adapter. The shim is the funnel — every error reaches the A2A
client as a valid JSON-RPC error response or SSE error event.

### A2A error names and code namespace

The A2A spec defines error **names** (string identifiers like
`TaskNotFoundError`) but does **NOT** assign canonical code numbers to
them — implementations are free to pick numbers within JSON-RPC's
reserved `-32000` to `-32099` server-error range. **A2A clients should
match on the error `name` field (and `data.kind`), not on the numeric
code.**

This shim assigns the following codes, but they are a sesh
implementation choice; other A2A implementations may use different
numbers for the same names:

| Sesh-assigned code | A2A name | When used |
|---|---|---|
| -32700 | Parse error | Malformed JSON in request body (JSON-RPC standard) |
| -32600 | Invalid Request | Request structure violates JSON-RPC spec (standard) |
| -32601 | Method not found | Unknown A2A method (standard) |
| -32602 | Invalid params | A2A method called with bad/missing params (standard) |
| -32603 | Internal error | Catch-all (standard) |
| -32001 | TaskNotFoundError | unknown taskId |
| -32002 | TaskNotCancelableError | `CancelTask` on terminal task |
| -32003 | PushNotificationNotSupportedError | shim has push disabled |
| -32004 | UnsupportedOperationError | adapter doesn't support operation |
| -32005 | ContentTypeNotSupportedError | adapter can't process media type |
| -32006 | InvalidAgentResponseError | adapter output untranslatable |
| -32007 | AuthenticatedExtendedCardNotConfiguredError | extended card unavailable |
| -32008 | ExtensionSupportRequiredError | adapter requires an extension the client didn't acknowledge |
| -32009 | VersionNotSupportedError | A2A protocol version mismatch |

Codes `-32008` and `-32009` correspond to A2A spec additions noted in
the a2a-go CHANGELOG (2026-02-23).

### Error envelope on the NATS bus

Adapters return errors using NATS micro convention plus structured body:

```
HEADERS:
  Nats-Service-Error-Code: 32001
  Nats-Service-Error: TaskNotFoundError

BODY:
  {
    "code": -32001,
    "name": "TaskNotFoundError",
    "message": "task 01ARYZ6S41TSRCOSKF2FVG69G2 not found in scope session:rc-dev",
    "data": {
      "taskId": "01ARYZ6S41TSRCOSKF2FVG69G2",
      "scope": {"kind": "session", "id": "rc-dev"},
      "retryable": false,
      "sesh.adapter": "claude",
      "sesh.machine": "cc",
      "sesh.role": "implementer"
    }
  }
```

Body matches A2A JSON-RPC `error` object shape. Shim copies it nearly
verbatim into the JSON-RPC response, stripping `sesh.*` fields from
`data` if the principal lacks the `sesh.observability` scope.

### Typed errors per origin

#### Adapter-originated

SDK provides typed helpers:

```typescript
throw new ContentTypeNotSupportedError("audio/wav not supported");
throw new InvalidAgentResponseError("model returned malformed tool call");
throw new UnsupportedOperationError("this agent does not support streaming");
```

Anything else becomes `-32603 Internal error` at the shim boundary,
with the original message preserved in `data.detail`.

#### Sesh-ops-originated

| sesh-ops error | A2A code |
|---|---|
| `task.ErrNotFound` | -32001 TaskNotFoundError |
| `task.ErrTerminal` (cancel on terminal) | -32002 TaskNotCancelableError |
| `task.ErrInvalidTransition` | -32602 Invalid params |
| `task.ErrCASConflict` | shim retries internally up to 3× with exponential backoff before surfacing; if exhausted: -32603 Internal error |
| `task.ErrSchemaViolation` | -32602 Invalid params |
| `task.ErrLeased` | -32603, `data.kind: "task-leased"`, `retryable: true` |

#### NATS-originated

| NATS condition | A2A code |
|---|---|
| No responder | -32601, `data.kind: "agent-not-present"` |
| Request timeout | -32603, `data.kind: "agent-timeout"`, `retryable: true` |
| Connection lost | HTTP 503 |
| JetStream KV unavailable | HTTP 503 |
| Subject permission denied | -32603, `data.kind: "permission-denied"` |

#### Shim-originated

| Shim error | Response |
|---|---|
| Auth missing | HTTP 401 |
| Auth invalid | HTTP 401 |
| Auth insufficient scope | -32603, `data.kind: "insufficient-scope"` |
| Rate limit exceeded | HTTP 429 with `Retry-After` |
| Signing key load failure | HTTP 500 (refuses startup) |
| Body too large | HTTP 413 |

### Errors during SSE streaming

Once SSE is open, errors emit as SSE events with `event: error`:

```
event: error
data: {"code":-32603,"name":"AdapterDisconnected","message":"adapter dropped mid-stream","data":{"taskId":"...","retryable":true}}
```

After a fatal SSE error, shim closes the stream. Client decides whether
to call `SubscribeToTask`.

**Important distinction**: a `TaskState=failed` is NOT an SSE error.
Task-failure is a normal terminal state, emitted as a
`TaskStatusUpdateEvent` with `status.state="TASK_STATE_FAILED"` (which
is terminal per `TaskState.Terminal()`). The Message inside
`status.message` carries the failure explanation. SSE `event: error`
is reserved for transport-level failures.

### Idempotency

A2A `SendMessage` is non-idempotent by default. Clients that need
idempotency supply a stable `messageId` (and optionally set
`message.referenceTaskIds` to bind to an existing Task).

**Define-out semantics** — duplicate `messageId` is not a client error:

- Shim attempts `Create` CAS on the Message. On CAS conflict (duplicate
  ID), shim reads the existing value:
  - **If existing body matches incoming body byte-for-byte** → return
    success silently with the existing Task object (idempotent retry).
  - **If existing body differs** → return `-32602 Invalid params` with
    `data.kind: "message-id-mismatch"` and a `data.detail` explaining
    the divergence.
- Task creation: `Create` CAS. Duplicate `taskId` returns the existing
  Task object (matching HTTP `PUT` semantics).

This pushes the burden of detecting "is this the same request" from
the client into the shim, where it has the durable state to make the
call. Clients can retry network failures freely.

### Retryability

`data.retryable` boolean signals whether to retry:

| Class | retryable | Notes |
|---|---|---|
| -32001 TaskNotFound | false | Retrying won't fix |
| -32002 TaskNotCancelable | false | Terminal state |
| -32602 Invalid params | false | Client must fix request |
| -32603, `concurrent-modification` | true | Backoff retry |
| -32603, `agent-timeout` | true | Bounded retries |
| -32603, `task-leased` | true | After `data.retry_after` seconds |
| -32603, `agent-not-present` | true (bounded) | Adapter may reconnect; ~30s window |
| -32004 / -32005 / -32006 | false | Capability mismatch; not transient |
| HTTP 503 | true | Standard infra retry |
| HTTP 429 | true | Honor `Retry-After` |

Shim adds `Sesh-Request-Id` response header on every response for
correlation.

### Logging vs propagation

| Detail level | Shim stdout | `data` (any auth) | `data` (observability scope) |
|---|---|---|---|
| `code`, `name`, `message` | yes | yes | yes |
| `data.kind`, `retryable`, `retry_after`, `taskId` | yes | yes | yes |
| `data.sesh.*` (adapter, machine, role) | yes | no | yes |
| Full Go error chain, stack traces | yes | no | no |
| KV revision numbers, NATS subjects | yes | no | yes |

### Adapter convention: state-failure vs RPC-error

Adapters should surface task-execution errors as Task transitioning to
`failed` (with explanatory Message in `status.message.parts[0].text`),
not as RPC-level errors. RPC errors are reserved for *immediate*
failures:

- Adapter doesn't support message content type → RPC `-32005`
- Task accepted then model errors mid-execution → Task → `failed` with
  Message explanation

This matches A2A client expectations.

### Observability for errors

```
sesh_shim_errors_total{code, kind, adapter}
sesh_shim_retryable_errors_total{kind}
sesh_adapter_rpc_errors_total{adapter, code}
```

Alert thresholds: high rate of -32603 `agent-not-present` indicates
adapter outage; high rate of -32603 `concurrent-modification` indicates
CAS contention.

### No sesh-specific error codes

Sesh-specific error context rides in `data.kind` string discriminators,
not in non-spec codes. Keeps shim output A2A-spec-pure and avoids
confusing A2A clients with hardcoded code mappings.

## Open questions deferred to per-repo plans

These were surfaced during design but don't block writing per-repo
plans. They should be revisited during the per-repo plan author's
Phase 3 (shim skeleton) and Phase 4/5 (SDK + full shim) implementation
plans:

1. **Skills authoring location**: per-adapter L3 (current
   recommendation) vs. central `sesh-skills/` config repo. The drift
   risk of per-adapter is acceptable but might surface ops pain at >20
   adapters.
2. **Notification dead-letter retention**: 7 days proposed. Make
   configurable in `shim.toml`.
3. **JWKS key rotation cadence**: not specified here. Ops will need a
   playbook for rotating the shim's signing key; either signed by an
   org CA with short-lived issuance, or operator-managed with manual
   rotation.
4. **Push notification webhook URL validation**: SSRF guardrails
   sketched in Section 7; final allowlist policy and DNS-rebinding
   defense decided in the shim implementation plan.
5. **Push notification credential encryption**: shim symmetric key vs
   KMS-managed key — operator policy decision; the shim implementation
   plan picks a default with a config knob.
6. **Sesh-ops library: atomic status+message helper signature**: final
   shape (parameter names, error returns, batch variant) decided in
   the sesh-ops plan.
7. **Multi-turn conversation semantics — `contextId` vs `taskId`**:
   per-turn-Task with `contextId` grouping is the recommendation; the
   shim plan formalizes the mapping.
8. **`ListTasks` filters and pagination**: shim plan defines the
   supported filter set (status, contextId, date range), pagination
   cursor shape, max page size.
9. **Cancellation-during-streaming behavior**: shim plan formalizes
   the sequence (KV write → KV watch fires → SSE emits cancelled
   status update → SSE closes → adapter sees KV change and interrupts).
10. **Shared contextId across principals in the same scope**: shim
    plan decides whether contextId is scope-shared or principal-private.
