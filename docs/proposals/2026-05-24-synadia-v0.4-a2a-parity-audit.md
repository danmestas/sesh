# v0.4 Design Audit

Deep audit of the v0.4 design doc against:

1. **The official a2a-go SDK** at `~/references/a2a-go/` (canonical source for protocol field names, wire shapes, and method signatures)
2. **Ousterhout's principles** (deep modules, information hiding, pull complexity down, define errors out of existence, design twice)
3. **Missing edge cases** from a protocol-completeness pass

Findings ranked by severity. Each finding is actionable — quoted fix included.

---

## Severity legend

| | Meaning |
|---|---|
| **P0** | Blocks correctness — the doc claims something that contradicts the protocol; ship-blocker |
| **P1** | Significant simplification — deletes whole categories of architecture without losing function |
| **P2** | Must address before Phase 5 (shim implementation) — edge cases, security, semantics |
| **P3** | Quality-of-life — typed builders, naming polish, single source of truth concerns |

## P0 — Protocol fidelity errors

### P0.1 — TaskState wire format is SCREAMING_SNAKE_CASE, not hyphenated lowercase

**SDK ref**: `a2a/core.go:279-301`. Actual values: `TASK_STATE_SUBMITTED`, `TASK_STATE_WORKING`, `TASK_STATE_INPUT_REQUIRED`, `TASK_STATE_AUTH_REQUIRED`, `TASK_STATE_COMPLETED`, `TASK_STATE_CANCELED`, `TASK_STATE_FAILED`, `TASK_STATE_REJECTED`, `TASK_STATE_UNSPECIFIED`.

**Doc claims** (Section 5): `submitted`, `working`, `input-required`, etc.

**Fix**: Update Section 5 wire-format tables. The internal sesh-ops Status enum can remain lowercase (`pending`, `in_progress`, …) — translation happens at the shim boundary. Add an explicit "shim translation" subsection showing the bidirectional mapping.

### P0.2 — Part has no `kind` discriminator; discrimination is by field presence

**SDK ref**: `a2a/core.go:582-756`. The `Part` struct has a polymorphic `Content` field; JSON marshaling flattens to `{text: "..."}` for text, `{data: {...}}` for data, `{url: "..."}` for URL ref, `{raw: "<base64>"}` for inline bytes.

**Doc claims** (Sections 2, 4): `{"kind": "text", "text": "..."}` and `{"kind": "file", "file": {"uri": ...}}`.

**Fix**: Drop the `kind` field everywhere it appears in Section 2's envelope sample and Section 4's value schemas. Replace with the flat shape:

```json
{"text": "Review this diff", "mediaType": "text/plain"}
{"url": "obj://...", "mediaType": "text/x-diff", "filename": "pr-1234.diff"}
{"data": {"foo": "bar"}, "mediaType": "application/json"}
{"raw": "<base64>", "mediaType": "application/pdf", "filename": "doc.pdf"}
```

Note: the SDK uses `filename` (not `name`), and supports a `raw` variant for inline bytes (also missing from our doc).

### P0.3 — `TaskStatusUpdateEvent.final` field does NOT exist

**SDK ref**: `a2a/core.go:516-562`. CHANGELOG (v1.0.0-alpha, 2026-02-23): *"`Final` field removed from `TaskStatusUpdateEvent`, state is used to determine whether event is final."*

**Doc claims** (Sections 5, 7, 8): `final: true|false` boolean on every `TaskStatusUpdateEvent`.

**Fix**: Remove `final` from all SSE event examples. Clients determine finality by checking `state.Terminal()`. This is also an Ousterhout win — defines out an error class (mismatch between `final` and `state`).

### P0.4 — SendMessageConfig field is `returnImmediately`, not `blocking` (and semantics inverted)

**SDK ref**: `a2a/core.go:768-781`. Field is `returnImmediately bool` (default false = block until terminal).

**Doc claims** (Section 7): `configuration.blocking: true`.

**Fix**: Rename to `returnImmediately` and flip the example. Default behavior is blocking (matches SDK).

### P0.5 — `AgentCard.url` is wrong — actual field is `supportedInterfaces[]`

**SDK ref**: `a2a/agent.go`. Required field is `SupportedInterfaces []*AgentInterface`; each `AgentInterface` has `URL`, `ProtocolBinding`, optional `Tenant`, optional `ProtocolVersion`.

**Doc claims** (Section 6, L1 layer): single `url` template populated from shim.toml.

**Fix**: L1 contributes `supportedInterfaces` array, typically with one entry (the shim's HTTPS+JSON-RPC endpoint). Multi-transport future is now naturally accommodated.

### P0.6 — `AgentCapabilities.stateTransitionHistory` doesn't exist in the SDK

**SDK ref**: `a2a/agent.go`. Actual fields: `extensions[]`, `pushNotifications`, `streaming`, `extendedAgentCard`.

**Doc claims** (Section 6): `streaming`, `pushNotifications`, `stateTransitionHistory`.

**Fix**: Replace `stateTransitionHistory` with the actual two missing fields: `extensions[]` (protocol extensions; each has uri/description/params/required) and `extendedAgentCard` (boolean indicating agent supports `GetExtendedAgentCard`).

### P0.7 — Two RPC methods missing entirely: `SubscribeToTask` and `ListTasks`

**SDK ref**: `a2asrv/handler.go:46, 55`. `SubscribeToTask` is the resubscribe primitive (live + history); `ListTasks` enumerates tasks within scope (with filters).

**Doc claims**: lists 11 RPC methods, omits these two.

**Fix**: Add both to the executive summary and to Section 7 (request lifecycle). `ListTasks` needs explicit auth-scoping semantics (Section 7) — principal-to-scope authorization gates what's returned. `SubscribeToTask` is the actual A2A name for what the doc calls `tasks/resubscribe`; verify the spec's method-naming convention and update consistently.

### P0.8 — Error codes `-32001..-32007` are not standardized

**SDK ref**: no hardcoded error codes in the SDK. The spec defines error NAMES (`TaskNotFoundError`, etc.) but not canonical numbers. CHANGELOG mentions new error names (`ExtensionSupportRequiredError`, `VersionNotSupportedError`).

**Doc claims** (Section 9): specific codes `-32001` through `-32007`.

**Fix**: Reframe Section 9. The shim defines its own code numbers (sesh implementation detail), but A2A clients should match on error `name` and `data.kind`, not on code number. Add `ExtensionSupportRequiredError` and `VersionNotSupportedError` to the error table.

### P0.9 — `Message.taskId` and `Message.contextId` are optional, not required

**SDK ref**: `a2a/core.go:199-227`. Both have `omitempty`. First-message-of-a-new-task has neither populated; the server assigns IDs in the response.

**Doc claims** (Section 2): both required in the envelope sample.

**Fix**: Update Section 2's envelope shape to show both as optional. Document the lifecycle:
- Client may omit both on initial `message/send`; shim assigns `taskId` and returns it
- Client supplies both on follow-up messages to continue the conversation
- `contextId` semantics: see P2.3

### P0.10 — RPC method naming convention is transport-specific

**SDK ref**: `a2asrv/handler.go:40-74`. JSON-RPC binding uses PascalCase (`SendMessage`, `GetTask`); REST binding uses `POST /message:send`.

**Doc claims**: REST-style throughout (`message/send`, `tasks/get`).

**Fix**: State the transport binding once at the top of Section 7 ("This shim implements the JSON-RPC binding; method names are `SendMessage`, `GetTask`, …") and use those names. Or note both forms in a translation table and pick one.

## P0 — Architecture fidelity errors

### P0.11 — Sesh-ops-served NATS subjects (`agents.task.add|get|list|cancel|subscribe`) are unnecessary pass-throughs

**Ousterhout principle**: pass-through methods are shallow by definition. They add abstraction cost without hiding meaningful complexity.

**Current design**: The shim calls `agents.task.add.*` (request/reply over NATS micro) → sesh-ops daemon receives → sesh-ops writes to KV.

**Reality**: KV is already CAS-protected. State-machine transition rules can be enforced at write time by a Go validation function. There's no behavior in sesh-ops's RPC handler that couldn't be in the shim itself, importing sesh-ops as a **library** (not calling it as a daemon).

**Implications if we make sesh-ops a library, not a daemon**:
- Delete the entire "Sesh-ops-served" subject category (5 subjects) from Section 3
- Sesh-ops repo ships a Go package, not a binary
- The shim imports `github.com/.../sesh-ops/task` and `github.com/.../sesh-ops/messages` etc.
- Lazy bucket creation moves into the shim (or into the sesh-ops package; either works)
- Other consumers (CLI, internal tools) ALSO import sesh-ops as a library — no daemon RPC needed

**Question to confirm with the user**: Is sesh-ops currently a long-running daemon, or is it already a library/CLI? This audit assumes library; if it's a daemon already, that's a different architectural starting point and the question becomes "do we want to add NATS RPC handlers to an existing daemon, or migrate it to a library?"

**Fix**: Either rewrite Section 3's sesh-ops subject category to "(removed; sesh-ops is a library)" or, if sesh-ops is genuinely a daemon, justify why each of those 5 RPC handlers is needed beyond the KV write itself. My recommendation is the former.

## P1 — Significant Ousterhout simplifications

### P1.1 — Drop `agents.notify.*` subjects (4 subjects)

**Red flag**: pass-through methods. The shim is the only caller; the shim writes to `sesh_notifyconfigs_*` KV bucket directly. The intermediate NATS subject adds no value.

**Fix**: Section 3 — remove the `agents.notify.{create,get,list,delete}` rows from the "Shim-served" table (which then becomes empty and can also be deleted). Section 7's push notification path becomes "shim writes to KV directly."

### P1.2 — Drop `sesh_cards_<scope>` cache bucket

**Red flag**: two-level caching with no clear win. The shim already has an in-process LRU cache for composed AgentCards with a 5-minute TTL. Adding a KV bucket as a "warm cache" means **two cache layers to invalidate**, two failure modes, and asymmetric staleness between shim replicas.

**Fix**: Section 4 — remove `sesh_cards_*` from the bucket table. Section 6 — remove the "optional optimization" paragraph. Shim fetches L3 contributions via `agents.card.get.*` request/reply, caches in-process, refreshes on TTL expiry or on $SRV.PING failure for the source agent.

### P1.3 — Drop `agents.task.subscribe.*` (one subject)

**Same red flag** as P0.11 — pass-through to a KV watch the caller could open directly. Shim and other consumers already KV-watch the Tasks bucket.

**Fix**: Section 3 — remove the row. Section 4's resubscribe mechanism already uses KV watch directly.

### P1.4 — Collapse AgentCard L1+L2+L3 → L1+L2 (shim defaults + adapter card)

**Red flag**: three layers of config a contributor must understand. Currently:
- L1 — shim.toml (operator config)
- L2 — `$SRV.INFO` metadata (legacy v0.3 advertising)
- L3 — `agents.card.get.*` responder (rich card)

The L2 layer exists primarily for "v0.3 adapter without L3 still gets a skeleton card." That's a degradation path, not a merge-input. Simpler model:

- **L1 — operator defaults** (always applied)
- **L2 — adapter contribution** (preferred: rich `agents.card.get.*`; fallback: skeleton from `$SRV.INFO`)

Same end-state, fewer ingredients, less mental load on contributors.

**Fix**: Rewrite Section 6's "Three-layer composition" subsection. Show the fallback explicitly as part of L2 resolution, not as a separate layer.

### P1.5 — Drop the `unknown` TaskState entirely

**Define-error-out**: My doc adds `unknown` as a "wire-only fallback when scope unresolvable." But by the time the shim is composing a response for `tasks/get`, scope is already resolved (otherwise the lookup couldn't have happened). The state is unreachable.

**Fix**: Section 5 — drop `unknown` from the table. If scope is unresolvable, return `TaskNotFoundError` (the actual canonical answer). If a Task exists in KV with a state we can't recognize, that's a schema error — log and return internal error. Don't invent phantom states.

### P1.6 — Drop `agents.event.*` from v0.4 scope (defer to its own design)

**Strategic programming**: `agents.event.*` is a sesh-wide lifecycle eventing system. It's not needed for A2A parity. A2A's event model is per-task `TaskStatusUpdateEvent`/`TaskArtifactUpdateEvent` (already covered). Mixing sesh-events into the A2A parity work conflates concerns and adds bus subjects nobody downstream demands.

**Fix**: Section 3 — remove the `agents.event.*` row. Add a brief note: "Sesh-wide lifecycle events (`session-start`, `tool-use-pre`, etc.) are outside the A2A parity surface. They will be designed separately if/when a consumer requires them."

### P1.7 — Drop `agents.shutdown.*` from v0.4 scope (or specify it)

**Red flag**: Half-finished feature. The verb appears in Section 3 but is never explained anywhere — what does graceful shutdown semantically mean for an adapter? Who's the caller? What's the response?

**Fix**: Either (a) drop the subject; or (b) specify in a Section 3 subsection: what triggers shutdown, what the adapter does (drain in-flight tasks? reject new prompts? exit process?), what the response shape is. My recommendation: drop. The mesh already has heartbeat-loss detection; adapters that need explicit shutdown can use OS signals.

### P1.8 — Merge `sesh_notifyconfigs_*` and `sesh_notifyfailed_*` into one bucket

**Red flag**: change amplification. Two buckets for one feature means any query of "give me healthy configs" requires joining. Better: one bucket with a status field on each config; failed-delivery history as a sub-collection or revision history.

**Fix**: Section 4 — collapse to a single `sesh_notifyconfigs_*` bucket. Config value gets a `delivery_status` and a `recent_failures` field. Optional separate bucket for archival can be added later if ops needs it.

### P1.9 — Drop `agents.card.extended.*` as a separate subject (or use it instead of inline `extended`)

**Red flag**: leaky abstraction — Section 6 has adapters return an L3 response with an inline `extended: {...}` field that the shim has to split. But Section 3 also lists `agents.card.extended.*` as a separate subject. Pick one mechanism:

- **Option A** (current): adapters return one big object; shim splits at HTTPS boundary. Inline `extended` field is a shim-side filter contract leaked into the adapter response shape.
- **Option B** (cleaner): adapters implement two NATS micro endpoints (`agents.card.get.*` and `agents.card.extended.*`). Shim calls one or the other depending on auth state. No inline filtering contract.

**Fix**: Pick B. Section 6 — remove the inline `extended` field. Section 3 — keep both subjects. Make adapter SDK helper register both handlers if extended is provided.

### P1.10 — Move `sesh.*` metadata out of Message body, into NATS headers

**Information hiding violation**: Section 4 puts `sesh.machine`, `sesh.role`, `sesh.inst` inside the A2A Message JSON. We claim "A2A consumers ignore unknown metadata per spec," but it's still on the wire and still in their logs and analytics. And the SHIM doesn't need it after routing — the routing already happened.

**Fix**: Sesh routing metadata travels as NATS message **headers** during transit, never enters the Message body. The Messages bucket value is **byte-for-byte A2A Message with no sesh-internal additions**. Truly invisible to A2A consumers. Mesh-internal observers can read NATS headers from the transit subject.

This is also a **pull-complexity-down** win: adapters don't have to remember to strip `sesh.*` keys before relaying; the cleanliness is structural.

## P2 — Missing edge cases (must address in Phase 5 plan)

### P2.1 — Duplicate `messageId` should be idempotent success, not -32602

**Define-error-out**: A2A clients may retry on network failure. If the retry sends the same `messageId` with the same body, that should succeed silently. Different body with same ID is a real conflict. My doc returns `-32602` for both, telling the client to "treat as success" — that's pushing complexity up.

**Fix**: Shim implements `Get`-then-`Create` semantics. If the existing message body matches, return success. If different, return a typed conflict error with `data.kind: "message-id-conflict"`. Section 9 should document this.

### P2.2 — Webhook URL validation for SSRF

**Security**: A malicious A2A client registers `http://internal-admin:8080/delete-all` as a webhook URL. Shim becomes a confused deputy.

**Fix**: Shim webhook delivery validates URLs:
- Reject schemes other than `https://` (configurable; `http://` allowed only for `localhost`/`127.0.0.1`/`::1` in dev mode)
- Reject hostnames resolving to private IP ranges (RFC 1918 + 169.254/16 + ULA fc00::/7 + loopback) — DNS rebinding protection: resolve at registration AND at delivery, reject if mismatch
- Optionally restrict to an operator-configured allowlist of webhook hostnames

Section 7's push notification path needs a dedicated subsection on this.

### P2.3 — `contextId` vs `taskId` semantics for multi-turn conversations

**Spec ambiguity**: A2A has `contextId` (conversation grouping) and `taskId` (task lifecycle). Multi-turn chat could be:
- One Task, many Messages (Task = conversation, Messages = turns)
- One Context, many Tasks (Context = conversation, each turn = new Task)

The SDK supports both interpretations; the spec doesn't mandate one. Sesh has to pick.

**Fix**: Decide and document in Section 5. My recommendation: **one Task per turn, contextId groups turns into a conversation**. Reasoning:
- Sesh-ops Task already has clear lifecycle (pending → in_progress → terminal). Mapping it to "the whole conversation" makes "terminal" awkward (when does a conversation end?).
- Per-turn Tasks let each turn have its own lifecycle, artifacts, retry semantics.
- `contextId` becomes the conversation-grouping key for client-side history reconstruction. We could surface a `sesh contextId` filter in `ListTasks` to fetch all turns.

### P2.4 — Cancellation during streaming

**Edge case**: A2A client opens `message/stream`, sees output streaming, sends `tasks/cancel` in a separate request. What happens?

**Fix**: Section 7 — define behavior:
- `tasks/cancel` writes `cancelled` to Tasks KV
- KV watcher fires on the active SSE
- Shim emits `TaskStatusUpdateEvent` with `state=TASK_STATE_CANCELED`, closes SSE
- Adapter, also watching KV (via its SDK), sees the cancellation and stops generating
- Any chunks the adapter was about to emit on `agents.task.stream.*` are dropped (no harm; SSE already closed)

### P2.5 — Push notification delivery semantics (at-least-once via durable consumer)

**Missing**: My doc says shim runs "background worker on KV watch." A naive KV watch is at-most-once — if the shim crashes between "watcher fires" and "POST succeeds," the notification is lost.

**Fix**: Push delivery uses a **JetStream durable consumer** on the Tasks KV stream, ack semantics, retry-on-nack. Shim worker pulls, attempts delivery, acks on success, nacks (with backoff) on failure. At-least-once delivery is the standard A2A expectation. Webhook receivers must be idempotent (they already need to be per spec). Section 7's push notification subsection needs this.

### P2.6 — Push notification config auth credentials at rest

**Security**: `PushConfig.Auth.Credentials` may be a bearer token, API key, or other secret. Plain JSON in KV means anyone with NATS read permission on the bucket can exfiltrate.

**Fix**: Encrypt-at-rest. Shim has a symmetric key (or uses cloud KMS); credentials field is `enc:<base64>` on write, decrypted only at delivery time. Key rotation playbook needed. Section 9 — add this constraint.

### P2.7 — Auth differentiation: public AgentCard vs extended card

**Spec**: `agent/getAuthenticatedExtendedCard` requires "auth". My doc says "successful authentication at HTTPS layer" — but the public AgentCard endpoint is also accessed over HTTPS (unauthenticated). How does the shim distinguish?

**Fix**: Add an OAuth2 scope (`agent.card.extended`) or a custom JWT claim. Public card endpoint requires no auth. Extended card endpoint requires (a) valid auth AND (b) the specific scope. Section 6 — add this differentiation. Section 7 — add scope validation to the auth middleware.

### P2.8 — `ListTasks` filter, pagination, and authorization semantics

**Missing**: Section 3 mentions `agents.task.list.*` (which P0.11 says we're removing as an RPC subject), but doesn't address what filters A2A's `ListTasks` accepts or how pagination works.

**Fix**: When implementing `ListTasks` in the shim (it reads from KV directly): document supported filters (status, contextId, created-after, created-before), pagination via cursor (KV key after which to start), maximum page size (cap at 100). Authorization: shim filters results to only those the principal can read.

### P2.9 — Lease contention during `message/send`

**Edge case**: Shim writes Task to KV; publishes prompt on coord subject. Two adapters in the queue group both attempt to lease the task (via sesh-ops's existing Puller semantics). Only one wins.

**Question**: Does the loser silently exit, or does it propagate an error somewhere?

**Fix**: Document existing sesh-ops behavior: Puller uses CAS lease acquisition; loser sees `ErrLeased`, drops the message silently (queue group already routed it to one). No error propagation to the A2A client — the system worked as designed.

### P2.10 — Concurrent `contextId` access

**Edge case**: Two principals both have access to scope `session:shared-team`. Both send messages with same `contextId`. Do they see each other's messages?

**Fix**: Decide. My recommendation: YES — `contextId` is shared across the principals authorized for the scope. This matches "shared session" intent. Otherwise, contextId is principal-scoped and we have a different security model. Section 7 — document.

## P3 — Quality of life

### P3.1 — Subject-name generation should be a typed builder

**Change amplification**: Section 3 documents subject patterns as strings. Adapter SDKs (TS), shim (Go), and sesh-ops (Go) all need to construct/parse these. Currently every consumer hand-writes string concatenation. If we ever change a subject layout, every callsite has to be updated.

**Fix**: Define typed subject builders:
- Go: `internal/subject/subject.go` in sesh (or in a shared sesh-protocol repo)
- TS: corresponding module in sesh-channels SDK

```go
// Example:
subj := subject.PromptV2(machine, project, session, role, inst)
parsed, err := subject.ParsePromptV2(subj)
```

The design doc references these builders rather than tabulating raw strings. The protocol spec lives in the type signatures.

### P3.2 — Shim internal CAS retry, not A2A-client-facing

**Pull complexity down**: My doc returns `-32603 concurrent-modification, retryable: true` to A2A clients on CAS conflict. But A2A clients don't have specific retry logic for sesh internals. Push the retry into the shim: 3 internal attempts with exponential backoff before surfacing any error.

**Fix**: Section 9 — shim-internal retry on CAS conflict. Only surface as A2A error if all retries exhausted. Error code becomes plain `-32603 internal error`, not `concurrent-modification` (which leaks our storage backend to the client).

### P3.3 — Scope-id sanitization: single source of truth

**Pull complexity down**: Section 3 says the rule is "applied identically by sesh-ops and shim." Two implementations risk drift.

**Fix**: One Go package (`internal/scope/sanitize.go`) imported by both. If sesh-ops is a library (per P0.11), this is natural — the function lives there.

### P3.4 — Object Store key shape: `<task-id>/<artifact-id>` not `<task-id>/<artifact-id>/<part-index>`

**Inconsistency**: Section 4 says "store the FINAL assembled artifact" in Object Store. But the key includes `<part-index>`, implying chunked storage. Pick one.

**Fix**: If chunks travel ephemerally over core NATS and only final artifacts persist, drop the `<part-index>` from the Object Store key. If we want resumable upload (chunked persistence), keep the part-index and document the assembly/finalize flow.

### P3.5 — Phase 4 is six features in a trench coat

**Change amplification within the rollout**: Phase 4 ships envelope-v2 detection + Messages writer + Artifacts writer + stream publisher + card helper + capability advertising — all in one PR? Or one milestone?

**Fix**: Section 8 — split Phase 4 into 4a (envelope-v2 detection + Messages writer) and 4b (Artifacts + stream + card + advertising). Each releases as its own SDK minor version. Adapters opt into 4a alone if they want chat-only A2A parity without artifact streaming.

## Cross-cutting recommendation: sesh-ops as library vs daemon

This audit's largest single architectural question — and the one that re-shapes Section 3 the most — is whether sesh-ops is a **library** imported by shim/sesh/etc., or a **long-running daemon** with its own NATS handlers.

If **library**: delete the entire `agents.task.*` RPC category. Shim talks directly to KV. Sesh-ops ships Go modules. State-machine rules enforced in code at write time. Simplest architecture.

If **daemon**: keep the RPC subjects (or some of them) and justify each with behavior that genuinely needs a centralized enforcer beyond KV CAS.

**Action**: confirm sesh-ops's current shape (library or daemon) with the user before re-writing Section 3. If it's currently a library, this entire RPC tier is over-engineering and should be deleted. If it's currently a daemon, we still want to question whether each RPC handler is adding value beyond KV.

## Recommended plan-of-attack

### Before AI reviewers see it

Apply P0 fixes (protocol fidelity) — these are mechanical, no architectural debate:

1. P0.1 — TaskState wire format
2. P0.2 — Part shape
3. P0.3 — drop `final` field
4. P0.4 — rename to `returnImmediately`
5. P0.5 — `supportedInterfaces[]`
6. P0.6 — capability field corrections
7. P0.7 — add `SubscribeToTask`/`ListTasks`
8. P0.8 — error code framing
9. P0.9 — taskId/contextId optionality
10. P0.10 — pick a method-name binding

Also resolve P0.11 by checking with the user on sesh-ops's current shape.

### Defer to AI reviewer input

P1 simplifications are architectural and benefit from a second opinion — let the reviewers weigh in before deleting subjects or merging buckets. Frame them in the audit as "candidate simplifications" and let reviewers either ratify or push back.

### Resolve in per-repo implementation plans

P2 edge cases are best handled in the per-repo plans (Phase 4/5 specifically): the SDK plan covers idempotency, the shim plan covers SSRF/auth/push delivery, etc. Don't try to resolve in the design doc; reference the open list and let the plans solve them.

P3 quality items go into the per-repo plans as well — they're implementation hygiene, not architecture.
